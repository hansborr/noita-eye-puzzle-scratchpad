# Noita Game Data and Tooling

This document covers how Noita stores game data, how to extract assets, the seed/PRNG and determinism model, debug/dev-mode and Lua inspection, and the community tools most useful for code investigation (e.g. of the Eye Messages). It distinguishes primary/verifiable evidence from community lore, and tags non-trivial claims by confidence.

A skeptic's caveat that frames the whole document: the Eye Messages specifically are **not** stored as ordinary, extractable assets. They are generated by engine-internal (compiled) code, not by Lua, and the glyphs are drawn on the fly rather than loaded from a sprite file [confirmed] (https://noita.wiki.gg/wiki/Eye_Messages ; https://en.wikipedia.org/wiki/Noita_(video_game)). So the asset-extraction pipeline below is necessary background and applies to almost everything in the game — but it does **not** by itself yield the eye content. That required binary reverse engineering (see "Reverse engineering" below).

## 1. How Noita stores data

### 1.1 `data.wak` — the WizardPak archive

Most game assets ship inside a single archive, `data.wak`, in the install directory. It is an uncompressed virtual file tree ("WizardPak"). The format is documented on the wiki and reimplemented by at least three independent tools, which agree on the layout [confirmed].

Header (12 bytes used, plus padding), little-endian (https://noita.wiki.gg/wiki/Technical:_File_Formats):

```
struct WizardArchive {
    le u32 version;
    le u32 file_count;
    le u32 first_file;
    padding[4];
    WakFile files[];
}
```

Each file-table entry (https://noita.wiki.gg/wiki/Technical:_File_Formats):

```
struct WakFile {
    le u32 offset;     // location of file data
    le u32 size;       // file size in bytes
    le u32 name_len;   // filename length
    char  name[name_len];
    char  data[size] @ offset;
}
```

The format is **little-endian throughout**, which the wiki notes is unusual because Noita's *save* files are otherwise big-endian [confirmed] (https://noita.wiki.gg/wiki/Technical:_File_Formats). Independent implementations confirming this layout: UnWak (C#) (https://github.com/isJuhn/UnWak), wpak/luawak (Lua) (https://github.com/zatherz/wpak), and a Python gist (https://gist.github.com/RidgeX/e159bb7df97b2e18209aea2804a79d7a).

### 1.2 data.wak encryption

There is a genuine correction to a widely-repeated claim here. The community `noita-player` doc hedges that the mode is "OFB or CTR" (and notes "OFB and CTR are the same if your data is a single block") (https://github.com/noita-player/noitadocs/wiki/Reversing-data.wak), but the actual working implementations use **AES-128 in CTR mode** [confirmed]: UnWak ships an `Aes128CounterMode` (AES-ECB + counter + XOR, i.e. textbook CTR) (https://github.com/isJuhn/UnWak), and the RidgeX gist explicitly uses `AES.MODE_CTR` (https://gist.github.com/RidgeX/e159bb7df97b2e18209aea2804a79d7a). Treat "OFB" as imprecise; CTR is what the code does.

Key/IV derivation (corrects the popular "PRNG seeded with constant 123" framing) [confirmed, with one nuance]:

- The derivation uses a **Park-Miller / MINSTD LCG** (multiplier 16807, modulus 2^31−1), per UnWak's `WakRng` (multiplier 16807, modulus `0x7FFFFFFF`) (https://github.com/isJuhn/UnWak).
- Seeds are: **0** (key), **1** (header IV), **2147483646** (filetable IV), and the **file index** (per-file IV). Each seed is first offset by **+0x165EC8F** (23,456,911) before seeding the PRNG (UnWak `WakTypes.cs`/`WakDecryptor.GenerateIV`; corroborated by the RidgeX gist's `derive_key(0)` master key and `derive_key(index)` per-entry IV) (https://github.com/isJuhn/UnWak ; https://gist.github.com/RidgeX/e159bb7df97b2e18209aea2804a79d7a).
- The **123 / 0x7B** that the `noita-player` doc mentions is **not** the key-derivation seed. It is a value the engine AES-*encrypts* while *constructing* the WizardPak ("they AES encrypt the integer 123 (0x7b) when constructing WizardPak for whatever reason") (https://github.com/noita-player/noitadocs/wiki/Reversing-data.wak). The `noita-player` doc's looser "seed 123 for the header / file index for files" phrasing is the source of the popular confusion [likely — the two community sources differ in how they describe the header seed; UnWak/RidgeX (working code) should be preferred over the prose doc].

The 0x165EC8F offset constant is independently attested by both UnWak and the `noita-player` doc (https://github.com/noita-player/noitadocs/wiki/Reversing-data.wak), which is good cross-source corroboration.

### 1.3 `.salakieli` files (saves / state) — a separate scheme

Noita encrypts certain save/state files with the `.salakieli` extension ("salakieli" is Finnish for a secret/cipher language) using **AES-128-CTR with literal passphrase-derived key/IV strings** — distinct from the data.wak PRNG scheme [confirmed] (https://gist.github.com/RidgeX/e159bb7df97b2e18209aea2804a79d7a ; https://noita.wiki.gg/wiki/Technical:_File_Formats). The first 16 bytes of each passphrase serve as key and IV. Documented pairs (key / IV):

- `magic_numbers`: `KnowledgeIsTheHighestOfTheHighest` / `WhoWouldntGiveEverythingForTrueKnowledge`
- `world_state`: `TheTruthIsThatThereIsNothing` / `MoreValuableThanKnowledge`
- `_stats`: `SecretsOfTheAllSeeing` / `ThreeEyesAreWatchingYou`
- `player`: `WeSeeATrueSeekerOfKnowledge` / `YouAreSoCloseToBeingEnlightened`

(https://gist.github.com/RidgeX/e159bb7df97b2e18209aea2804a79d7a). The thematic phrasing ("all-seeing", "three eyes are watching you") is a flavor choice, not a clue to the Eye Messages cipher — do not over-read it [speculative-to-avoid].

### 1.4 Asset formats inside the archive

Once unpacked, Noita assets are conventional, human-readable formats [confirmed by the existence and routine use of unpackers; specific to most assets, not the eyes]:

- **PNG** for sprites/backgrounds (e.g. the spawn-gating background `background_cave_02.png` referenced by the eye-message generator) (https://noita.wiki.gg/wiki/Eye_Messages).
- **XML** for entity/component definitions, biomes, materials, etc.
- **Lua** scripts for most game logic and modding hooks.

Fonts/"alphabets": Noita uses bitmap-style fonts and several in-game symbol systems. Note the important distinction: several Noita symbol sets that *do* live in assets are decoded (e.g. the Common/English glyphs, Material Symbols, the Orb-Room/Finnish-creation-myth glyphs) (https://steamcommunity.com/sharedfiles/filedetails/?id=3281214266) — but the **Eye Messages have no associated sprite/font asset at all**; they are rendered procedurally by the engine [confirmed] (https://noita.fandom.com/wiki/Eye_Messages ; https://noita.wiki.gg/wiki/Eye_Messages). Any tool that decodes eye glyphs works from *transcriptions* of in-world renders, not from an extracted font.

## 2. Extracting assets

### 2.1 Official, no-third-party-tools method

Noita can unpack its own archive. Pass **`-wizard_unpak`** on the command line ("Used to extract the game files from `data.wak`") [confirmed] (https://noita.wiki.gg/wiki/Launch_Options ; https://noita.wiki.gg/wiki/Technical:_File_Formats). On a typical install the community workflow is to run the provided unpack helper from `tools_modding/` (e.g. `data_wak_unpack.bat` on Windows) from the game root [confirmed] (https://noita.wiki.gg/wiki/Technical:_File_Formats).

### 2.2 Community reimplementations

These reproduce the format independently and are useful when you want programmatic access or are not on Windows [confirmed they exist and parse the format]:

- **wpak / luawak** (Lua): `wpak.lua unpack data.wak some_directory/` — https://github.com/zatherz/wpak
- **UnWak** (C#): full WizardPak parse + AES-128-CTR decrypt — https://github.com/isJuhn/UnWak
- **RidgeX gist** (Python): handles both `.wak` and `.salakieli` — https://gist.github.com/RidgeX/e159bb7df97b2e18209aea2804a79d7a
- Additional unpackers referenced by the community: Noita-Wak-Extractor (dextercd) and a write-up "Decrypting Noita's files" (Dexter Döpping) (https://www.linkedin.com/pulse/decrypting-noitas-files-dexter-d%C3%B6pping) [likely; not independently re-verified here].

Reminder: none of these will surface the Eye Message content, since it isn't in the archive [confirmed] (https://noita.wiki.gg/wiki/Eye_Messages).

## 3. Seed, PRNG, and determinism

### 3.1 The PRNG

Noita uses a **Lewis-Goodman-Miller (LGM) generator** (i.e. a Lehmer/Park–Miller MINSTD-style multiplicative congruential generator), with a **31-bit state**, exposed via `SetRandomSeed(x, y)` and `Next()` [confirmed] (https://noita.wiki.gg/wiki/Technical:_Noita_PRNG). `SetRandomSeed` mixes the world seed with two numeric inputs (typically coordinates) to set global state; `Next()` advances state and returns a value scaled into a 64-bit float in (0,1) (https://noita.wiki.gg/wiki/Technical:_Noita_PRNG).

### 3.2 Determinism and placement

Determinism is **position-dependent**: "every random event in the game depends both on the world seed and the position the event happens at," and the NG+ count is added to the world seed for seeding purposes [confirmed] (https://noita.wiki.gg/wiki/Technical:_Noita_PRNG). The same seed therefore reproduces the same world.

For the Eye Messages specifically:

- **Spawn locations are seed-deterministic** [confirmed]: "a given seed will always spawn them in the same locations," and per-seed coordinate tables exist (e.g. seed 1249563923 lists East 1–5 and West 1–4 with mirrored X across the two parallel worlds) (https://noita.wiki.gg/wiki/Eye_Messages). Lymm's tooling reproduces these coordinates from a seed *without launching the game*, which is independent evidence of determinism (https://gitlab.com/realgonzogames/lymms-binoculars).
- **Whether the message CONTENT is byte-for-byte identical across all seeds is asserted by the community but not proven in a retrievable primary source** [likely, not confirmed]. Xkeeper0's transcoder hard-codes a single fixed set of message integers (implying seed-invariant content) (https://gist.githubusercontent.com/Xkeeper0/a6eda18571ef889be291822c400cc6c8/raw), and seed finders return only coordinates — but I found no primary source demonstrating content invariance across many independent seeds. Treat "content is the same on every seed" as plausible consensus, not verified fact.
- Additional spawn gates (all [confirmed], https://noita.wiki.gg/wiki/Eye_Messages): the `mods_have_been_active_during_this_run` flag must be unset (condition resets on game close); the "Entered East/West" parallel-world message must have triggered; and the location must use `background_cave_02.png`.

## 4. Debug/dev mode and Lua modding for inspection

- **`noita_dev.exe` (dev build)**: ships in `tools_modding/` and must be launched from the game root (next to `noita.exe`); it provides a development console and surfaces Lua/XML errors as they happen [confirmed] (https://steamcommunity.com/app/881100/discussions/0/3476150824940419212/ ; https://noita.wiki.gg/wiki/Modding:_Basics).
- **F5 debug features / console output**: F5 enables most in-game debug features; `print()` output is only visible with debug enabled, while `GamePrint()` renders in-game regardless [confirmed] (https://noita.fandom.com/wiki/Modding:_Basics).
- **`-debug_lua`**: "Enabled debugging hooks. Only works with noita_dev.exe" — needed for external Lua debuggers (Decoda, VS Code) [confirmed] (https://noita.wiki.gg/wiki/Launch_Options).
- **`-magic_numbers <path>`**: loads Magic Numbers from a file; notably this "does not count as playing modded," so achievements stay enabled. Pointing at a nonexistent path dumps all magic variables [confirmed] (https://noita.wiki.gg/wiki/Launch_Options). Relevant for inspection because it lets you tweak engine tunables without the mod flag that would otherwise *disable* eye-message spawning.
- **Component Explorer / `unsafe-explorer` mod**: enables, inside the Steam `noita.exe`, debug/inspection features normally exclusive to `noita_dev.exe` (browse entities/components live) [confirmed] (https://noita.wiki.gg/wiki/Mod:Component_Explorer).
- **VS Code Lua debugger**: Ismoh/noita-vscode-debugger — https://github.com/Ismoh/noita-vscode-debugger [confirmed it exists].
- **Critical inspection caveat for the eyes**: enabling *any* non-translation mod sets the mods-active flag and **prevents eye messages from generating** [confirmed] (https://noita.wiki.gg/wiki/Eye_Messages). So Lua/mod-based inspection is excellent for the rest of the game but is the wrong tool for capturing eye content; people instead use save-edited unmodded runs, Cheat Engine, or binary RE.

## 5. Reverse engineering the engine (how the eye content was actually obtained)

Because the eye generator is compiled into the engine (not Lua, not an asset), the content was recovered by binary reverse engineering, not by unpacking [confirmed]:

- **Ninji** (game reverse engineer `@_Ninji` / wuffs.org — distinct from "ghidraninja"/stacksmashing) disassembled a Noita build (community sources cite the Feb 3, 2021 build) and published a **Ghidra project (~60 MB)** plus a **decompiled message-generation function** [likely — the description is consistent across the canonical Google Doc and wiki, but I could not directly load the Ghidra project or verify the 60 MB figure] (https://docs.google.com/document/d/1s6gxrc1iLJ78iFfqC2d4qpB9_r_c5U5KwoHVYFFrjy0 ; https://wuffs.org/blog ; reverse-engineer identity context: https://github.com/Treeki/CylindricalEarth). A save file from FuryForged is credited as assisting this work [likely] (https://noita.fandom.com/wiki/Eye_Messages).
- **Xkeeper0's PHP transcoder** reproduces the engine's decode math and is a *primary, runnable artifact*: it hard-codes the 9 messages as arrays of `[u32, u32]` pairs, combines each pair into a 64-bit integer (`a = n0 + n1*2^32`), then does **base-7 extraction** (divide by 7 once, then repeatedly emit `(a mod 7) − 1` and divide by 7), yielding symbol values in −1..5 where 5 = newline [confirmed] (https://gist.githubusercontent.com/Xkeeper0/a6eda18571ef889be291822c400cc6c8/raw). This is an important layering point for code investigators: the **engine's internal storage/decode is base-7 over 64-bit ints**, whereas the familiar **base-5 trigram (000–444, 0–124) is the human reading of the rendered 5-orientation glyphs**. Both are true at different layers; conflating them is a common error.
- The eye *generation/placement* is thus fully recoverable from the binary, but this is **not** the same as solving the *cipher* — the messages remain unsolved with no public method-backed plaintext [confirmed] (https://noita.wiki.gg/wiki/Eye_Messages).

### Evidence-quality note
Two cautions for anyone building on this corpus. First, the entire eye-content pipeline traces back to a **small set of contributors and a handful of repos/Google Docs** (Ninji, Xkeeper, Lymm, Toboter, CodeWarrior0), so independent reproduction of the *binary* RE step is thin — most downstream tools share the same upstream transcription [confirmed as a structural fact about the corpus]. Cross-validate transcriptions across ngraham20, SirCapybar/Doctor-Ned, and ToboterXP before trusting any single one. Second, Nolla actively trolls dataminers (e.g. the official `noitagame.com/for_the_seekers_of_truest_of_knowledge/` URL 302-redirects to a Rickroll) [confirmed] (https://noitagame.com/for_the_seekers_of_truest_of_knowledge/ ; https://tvtropes.org/pmwiki/pmwiki.php/Trivia/Noita), which is a reason to be cautious about treating thematic strings or "dev confirmation" lore as load-bearing.
