# 03 - base-marginalized local recovery, `s = 2..3`

**Status:** first bounded instrument built 2026-07-07 on branch
`agent/gak-unknown-base-recovery-01`.

## Scope

This is the third rung of `gak-unknown-base-recovery/`: a bounded
known-plaintext local-search instrument for hidden-base top-card-swap keys with
swap budget `s = 2..3`.

It is not an eyes run and it does not score language. It consumes planted or
supplied known plaintext/ciphertext pairs and accepts a candidate only when the
recovered mapping re-encrypts the compressed ciphertext exactly.

Unlike task 02, this is not exhaustive over hidden bases. A miss is reported as
`SearchCapExceeded`, not as proof that no key exists.

## Exact Cipher Convention

The solver uses the same Lymm deck convention as tasks 01 and 02:

```text
compose(p1, p2)[i] = p2[p1[i]]
state = compose(perm(L), state)
emit  ct_alphabet[state[0]]
perm(L) = B o sigma_L
```

`sigma_L` is a de-duplicated final permutation generated by at most `s` top-card
swaps `(0,k)`. Each message starts from identity. The ciphertext in
`KnownPlaintextPair` is the compressed emission stream: one ciphertext symbol
per plaintext-alphabet event.

## Algorithm

The solver searches over per-letter `sigma_L` assignments rather than hidden
bases.

For one assignment:

1. infer a partial hidden base from identity-restart first-symbol anchors:
   `B[sigma_L[0]] = c_0(L)`;
2. fill unanchored base positions deterministically to get one representative
   base;
3. replay all known plaintext under `perm(L) = B o sigma_L`;
4. accept only when compressed replay has zero mismatches;
5. feed the representative mapping into `audit_hidden_base_mapping` for
   planted/equivalent/ambiguous classification.

The local search uses deterministic random restarts plus coordinate descent over
the de-duplicated `sigma` domain. For `s=3`, the objective also uses a cheap
second-symbol identity-restart consistency term:

```text
message starts L M, second emission y
anchor(M) = B[t_M]
y = B[sigma_L[anchor(M)]]
```

When `y` is itself an anchored value, this constrains
`top_source(owner(y)) = sigma_L[anchor(M)]`.

## Controls

The self-test exercises the same library surface as the CLI:

| control | expected | measured outcome |
| --- | --- | --- |
| planted `s=2`, `n=7` positive | exact recovery | recovered planted base, best mismatches `0` |
| planted `s=3`, `n=5` positive | exact recovery | recovered exact key; representative audit reports an equivalent-base class |
| ciphertext-label shuffle null | no exact key within budget | `SearchCapExceeded`, no exact key |
| over-budget `s=3` attacked as `s=2` | no exact key within budget | `SearchCapExceeded`, no exact key |

Focused test command:

```sh
cargo test --locked hidden_base_local -- --nocapture
```

## Measured Results

These are synthetic known-plaintext fixtures. They are model-conditional
measurements, not claims about the real eyes.

| command shape | state | search surface | result |
| --- | --- | --- | --- |
| `n=5, s=3, messages=6, len=32, attempts=256, rounds=24` | `AmbiguousEquivalentClass` | sigma domain `47`, brute-force `5! = 120` | exact `192/192`; planted base retained, but audit found `5` compatible bases |
| default controls | PASS | `s=2` positive, `s=3` small positive, shuffle null, over-budget null | positives exact; nulls no exact within budget |
| `n=7, s=2, messages=8, len=48, attempts=512, rounds=24` | `SearchCapExceeded` | sigma domain `37`, brute-force `7! = 5040` | best observed replay `118/384`; no exact key found |
| `n=7, s=3, messages=8, len=32, attempts=512, rounds=24` | `SearchCapExceeded` | sigma domain `172`, brute-force `7! = 5040` | best observed replay `147/256`; no exact key found |

Interpretation:

- The task-03 instrument now has exact positive controls beyond `s=1`, including
  a small `s=3` fixture where non-identifiability is measured rather than hidden.
- The bounded local search is not yet a general `n=7` solver. The current
  `n=7` misses are search-budget misses, not mathematical no-candidate results.
- Candidate-evaluation counts are not directly comparable to `n!`: one restart
  scores many `sigma` substitutions. The relevant comparison currently reported
  is restarts/evaluations against the hidden-base brute-force size, with exact
  re-encryption as the only success criterion.

## Limits

- This is a bounded heuristic. `SearchCapExceeded` means "no exact key found
  within this local-search budget."
- The representative base is inferred from first-symbol anchors; sparse or
  non-anchored corpora can leave large equivalent classes.
- The `s=3` positive currently uses `n=5`. Extending exact recovery to `n=7`
  remains open.
- No eyes corpus and no language-scored ciphertext-only attack was run.

## Next Rung

The immediate follow-up is to improve the task-03 search frontier rather than
move to ciphertext-only:

1. add a top-source CSP/beam layer before coordinate descent;
2. keep the exact replay and hidden-base audit gates unchanged;
3. measure whether `n=7, s=2` and then `n=7, s=3` move from
   `SearchCapExceeded` to exact recovery;
4. record misses as budgeted misses unless an exhaustive baseline proves
   `NoCandidate`.
